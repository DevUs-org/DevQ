DevQ Phase 4 sanity test plan
Run `python main.py` from the project root (3 devices: d0 random devq 7q, d1 FakeNairobiV2, d2 FakeLagosV2).
Config files are referenced by name; all live in config/config_examples/. A block's stated config
stays in effect for every following block until a block states a new one.
NOTES: d0's error map is random per launch — deterministic checks pin --exec=d1,d2 or exclude d0.
Aer counts are unseeded — check mappings/states/reasons, not counts. Job IDs assume this exact
order in a fresh session. d1/d2 calibration values assume qiskit-ibm-runtime 0.45.1 — verify
Block 1's qerrors output first; if those match, everything downstream should too.

Block format:  Block N ~ Instructions ~ Commands ~ Expectations ~

Block 10
~
Determinism / seeding (Phase 5.1). Same three devices and config as every other block
(router_only.config.json); the only difference is main.py's --seed flag, which passes a
seed to both providers at construction. Run the command list FOUR times as four separate
fresh launches, saving each session's output:
  Run A: python main.py --seed 42    Run B: python main.py --seed 42
  Run C: python main.py              Run D: python main.py
Runs A and B are the reproducibility check; Runs C and D are the unseeded negative control.
Diff the full transcripts A vs B, then C vs D. This is the ONLY block that asserts counts —
seeding is what makes them assertable. `python main.py --help` should list --seed.
~
qerrors q d0
qtopology d0
qrun test_circuits/bell.qasm --exec=d1
qrun test_circuits/bell.qasm --exec=d1
qrun test_circuits/bell.qasm --exec=d2
qps
~
Seeded reproducibility across launches. Run A and Run B transcripts are IDENTICAL
byte-for-byte (diff clean) — this is the headline assertion, covering both d0's generated
device and all three jobs' counts. Specifics, on qiskit-ibm-runtime 0.45.1:
d0 qerrors q under seed=42: 0->0.0352 1->0.0177 2->0.0086 3->0.0479 4->0.0175 5->0.0055
6->0.0057. d0 qtopology under seed=42: 0--2, 0--3, 0--4, 1--3, 1--5, 1--6, 2--4, 2--5,
3--5, 4--6 (10 edges). Job 1 -> d1 {0:1, 1:2} counts {'00':1004, '11':937, '10':58,
'01':49}. Job 2 -> d1 {0:1, 1:2} counts {'00':996, '11':954, '01':50, '10':48}. Job 3 ->
d2 {0:1, 1:3} counts {'00':871, '11':867, '01':165, '10':145}. All three FINISHED.
NEGATIVE CONTROL 1 (distinct runs stay distinct): Jobs 1 and 2 are the SAME circuit on the
SAME device and mapping, but their counts DIFFER — derived per-run seeds (seed+k), not a
single reused seed cloning results.
NEGATIVE CONTROL 2 (omitting --seed is unchanged): Runs C and D DIFFER from each other and
from Run A — unseeded behaviour is preserved exactly, so every other block in this file
still behaves as written. C and D's job mappings still match Blocks 1-2 ({0:1,1:2} on d1,
{0:1,1:3} on d2); only counts and d0's generated device vary between them.
BUG-FIX WITNESSES (Phase 5.1 fixed two execution bugs; these numbers only hold after both):
Job 1's error weight outside 00/11 is ~5%, NOT ~27% — d1 executes under NAIROBI's noise
model, not Lagos's (fixed shared-noise-model leak). And ~5% rather than ~10% reflects the
allocator's mapping {0:1, 1:2} actually reaching the simulator as initial_layout=[1,2]
(fixed dropped v2p_map). Job 3's ~15% error on Lagos {0:1, 1:3} is likewise real: physical
qubit 1's 13.6% readout error dominates. If any of these three error rates is far off,
suspect the pinned stack before the code — verify Block 1's calibration output first.
